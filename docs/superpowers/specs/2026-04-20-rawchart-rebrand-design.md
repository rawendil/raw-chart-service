# RawChart Rebrand — Design

**Date:** 2026-04-20
**Status:** Draft
**Scope:** Rename the project from "Chart Service" / `chart-service` to "RawChart" / `rawchart-service` across user-facing text and technical identifiers. API paths, repo directory, and Git remote remain unchanged.

## Goals

- Present the product as **RawChart** in all surfaces users see (README, Swagger title, HTTP root response, logs).
- Update technical identifiers (npm package name, Docker image tag, Kubernetes resource names, container names, network name, database name) to the consistent slug `rawchart-service` (or `rawchart_service` where a snake_case identifier is required).
- Fix a latent inconsistency between `kubernetes/deployment.yml` and `docker-compose.yml` around the Postgres database name (see "Incidental fix" below).

## Non-goals

- **API paths** (`/api/charts/*`) remain unchanged. They describe the resource, not the brand.
- **Database table names** (`charts`, `chart_access_logs`) remain unchanged.
- **Repository directory name** and **Git remote** remain unchanged.
- **Logo and favicons** (`logo.svg`, `favicon.*`, `apple-touch-icon.png`, `site.webmanifest`) are intentionally out of scope. They will be regenerated from a new logo in a separate task.
- **Historical docs** under `docs/superpowers/plans/` and `docs/superpowers/specs/` are treated as historical artifacts and not rewritten.
- **README migration guide** for existing deployments is not included — the project is always treated as a fresh deployment.

## Naming conventions

| Context | Value |
|---|---|
| Brand (user-facing) | `RawChart` |
| Technical slug (kebab-case) | `rawchart-service` |
| Database / env slug (snake_case) | `rawchart_service` |

## Files to change

### User-facing text

- **`README.md`** — title heading, descriptive prose, `cd chart-service` → `cd rawchart-service`, `docker build -t chart-service` → `docker build -t rawchart-service`, and any remaining occurrences of "Chart Service" → "RawChart".
- **`src/index.ts`** — `customSiteTitle: 'Chart Service API Documentation'` → `'RawChart API Documentation'`; OpenAPI JSDoc description for `GET /` mentioning "Chart Service" → "RawChart"; JSON `message: 'Chart Service API'` → `'RawChart API'`.
- **`src/config/swagger.ts`** — `title: 'Chart Service API'` → `'RawChart API'`.
- **`src/utils/logger.ts`** — `defaultMeta: { service: 'chart-service' }` → `{ service: 'rawchart-service' }`.

### Technical identifiers

- **`package.json`** — `"name": "chart-service"` → `"rawchart-service"`; `docker:build` and `docker:run` scripts use `rawchart-service` image tag.
- **`package-lock.json`** — regenerated by running `npm install` after editing `package.json` (not edited by hand).
- **`docker-compose.yml`** — `container_name` for postgres / redis / app (`chart-service-db`, `chart-service-redis`, `chart-service-app`) → `rawchart-service-db`, `rawchart-service-redis`, `rawchart-service-app`; `POSTGRES_DB: chart_service` → `rawchart_service`; healthcheck `-d chart_service` → `-d rawchart_service`; `DB_NAME: chart_service` → `rawchart_service`; `networks.name: chart-service-network` → `rawchart-service-network`; any `# Chart Service Application` comment updated accordingly.
- **`kubernetes/deployment.yml`** — every `name:` / `app:` / container `image:` / secret reference (`chart-service-secrets`) / configmap reference (`chart-service-config`) / ingress (`chart-service-ingress`, `chart-service-tls`) / service backend name updated to the `rawchart-service` slug. **`DB_NAME` ConfigMap value** is set to `rawchart_service`.
- **`.env.example`** — `DB_NAME=chart_service` → `DB_NAME=rawchart_service`.

### Incidental fix

`kubernetes/deployment.yml` line 153 currently defines `DB_NAME: "chart-service"` (kebab-case), while `docker-compose.yml` uses `chart_service` (snake_case). Postgres database names are case/character-sensitive, so these do not refer to the same database — this is a latent configuration bug. Setting both to `rawchart_service` as part of this rebrand brings them into alignment. This is explicitly scoped in because it falls directly on the lines we are already touching; we are not expanding into unrelated cleanup.

## Files intentionally not changed

- `logo.svg`, `favicon.ico`, `favicon.svg`, `favicon-96x96.png`, `apple-touch-icon.png`, `site.webmanifest` — regenerated in a separate task from a new logo.
- `docs/superpowers/plans/*.md`, `docs/superpowers/specs/2026-04-20-cleanup-jwt-and-zod-migration-design.md` — historical artifacts.
- `src/routes/charts.ts`, `src/services/chartGenerator.ts`, `src/services/database.ts` (table DDL), and any other code that references the resource name "chart(s)" rather than the product name "Chart Service".
- `LICENSE` — unchanged unless the copyright holder line happens to mention the project name (to be verified while editing; the spec does not presuppose an edit).

## Verification

After edits, the following commands must pass:

1. **Residual-references check:**
   ```
   grep -rn "chart-service\|chart_service\|Chart Service" \
     --exclude-dir=node_modules \
     --exclude-dir=.git \
     --exclude=package-lock.json \
     .
   ```
   Expected output: matches only inside `docs/superpowers/` (historical specs/plans) and possibly inside `LICENSE` if the copyright line intentionally uses the old brand. Any other match is a miss.

2. **Build:** `npm run build` succeeds.

3. **Smoke test (manual, optional):**
   - `podman compose up --build` starts all three containers.
   - `curl http://localhost:3000/` returns a JSON body containing `"message": "RawChart API"`.
   - `curl http://localhost:3000/api-docs/` serves the Swagger UI with title **"RawChart API Documentation"**.

## Risks

- **`package-lock.json` drift:** changing `name` in `package.json` requires regenerating the lockfile. If skipped, `npm ci` on CI may complain. Mitigation: run `npm install` as part of the implementation and commit the updated lockfile.
- **Stale Postgres volume (dev machines):** anyone who previously ran the project will have a `postgres_data` volume containing a database named `chart_service`. With the new `POSTGRES_DB=rawchart_service`, Postgres will not auto-create the new database on an already-initialised volume, and the app will fail to connect. Per project decision, no README migration guide is provided — affected developers simply remove the volume (`podman compose down -v`) and start fresh.
- **K8s rollout:** changing Secret/ConfigMap names requires re-applying all manifests; a rolling update against only the Deployment would fail to find renamed Secrets. Scope note for the implementation plan: `kubectl apply -f kubernetes/deployment.yml` applies the full bundle, which is the expected deployment path.
